## 1. 软件概述

CPU Usage Track 是一款 Windows 桌面应用，用于实时监控系统 CPU 使用率。

**核心能力：**

| 能力 | 说明 |
|------|------|
| 实时曲线 | pyqtgraph 绘制 CPU 使用率随时间变化的曲线 |
| 智能告警 | CPU 超过设定阈值时自动抓取 Top 10 进程并生成日志 |
| 手动快照 | 一键抓取当前 Top 20 进程 CPU 占用详情 |
| 数据导出 | 将监控数据保存为 CSV 文件 |
| 数据精准 | 使用 Windows PDH Processor Utility，与任务管理器数据一致 |
| 极低开销 | 常规运行 CPU 占用 < 0.5%，不干扰被监控系统 |

## 2. 安装与启动

### 2.1 从源码运行

**环境要求：** Python 3.10+，Windows 10 1903+ / Windows 11


## 4. 基本操作

### 4.1 开始监控

1. 点击 **▶ 开始**
2. 曲线图开始实时绘制 CPU 使用率
3. 状态栏显示当前 CPU 值和运行时长
4. CPU 超过告警阈值时自动触发告警

### 4.2 暂停与恢复

1. 运行中点击 **⏸ 中断** → 暂停数据采集，曲线冻结
2. 暂停后按钮变为 **▶ 恢复** → 点击恢复采集
3. 运行时长会精确扣除暂停期间的时间

### 4.3 停止监控

1. 点击 **⏹ 停止** 结束本次监控
2. 停止后仍可点击 **💾 保存** 导出已采集的数据
3. 再次点击 **▶ 开始** 会清空旧数据，重新开始

### 4.4 关闭窗口

直接关闭窗口即可，应用会自动优雅地停止监控线程并释放资源。

## 5. 告警系统

### 5.1 触发条件

当 CPU 使用率 **≥ 告警阈值** 时自动触发，无需手动操作。

### 5.2 触发流程

1. 检测到 CPU ≥ 阈值 且 距上次告警 ≥ 30 秒（冷却期）
2. 启动一次独立的 1 秒采样窗口
3. 在采样窗口内同时获取**总体 CPU** 和 **Top 10 进程**
4. 按比例分配，确保进程 CPU 累加值 = 总体 CPU 值
5. 生成 Markdown 告警日志，保存到 `logs/` 目录
6. 在界面告警列表中新增一条红色记录

### 5.3 防刷机制

连续两次告警之间最少间隔 **30 秒**（`log_cooldown`），防止 CPU 持续高位时疯狂刷日志。

### 5.4 告警日志格式

文件命名：`cpu_alert_YYYYMMDD_HHMMSS.md`

```markdown
# CPU 告警日志

- **时间**: 2026-04-22 14:32:05
- **CPU 使用率**: 95.20%
- **告警阈值**: 已超过设定阈值

## CPU 占用率 Top 10 进程

| 排名 | 进程名 | PID | CPU 占用率 |
|------|--------|-----|-----------|
| 1 | chrome.exe | 1234 | 45.20% |
| 2 | devenv.exe | 5678 | 22.10% |
| ... | ... | ... | ... |

---
*由 CPU Usage Track 自动生成*
```

> 📌 日志中的"CPU 使用率"和 Top 10 进程列表来自**同一个采样窗口**，数据完全自洽——Top 10 累加值精确等于头部显示的总体 CPU 值。

## 6. 手动快照

### 6.1 用途

需要查看当前系统进程 CPU 占用明细时，手动触发一次快照。

### 6.2 操作方法

1. 点击 **📋 快照** 按钮
2. 按钮变为"📋 采集中…"，后台启动独立线程采集（约 1 秒）
3. 采集完成后，生成包含 **Top 20 进程**的 Markdown 日志
4. 告警列表中新增一条紫色记录
5. 双击条目可打开日志文件

### 6.3 快照日志格式

文件命名：`cpu_snapshot_YYYYMMDD_HHMMSS.md`

```markdown
# CPU 进程快照

- **时间**: 2026-04-22 14:35:10
- **CPU 使用率**: 12.50%

## CPU 占用率 Top 20 进程

| 排名 | 进程名 | PID | CPU 占用率 |
|------|--------|-----|-----------|
| 1 | rider64.exe | 12748 | 3.20% |
| 2 | chrome.exe | 1234 | 2.10% |
| ... | ... | ... | ... |

---
*由 CPU Usage Track 手动快照生成*
```

### 6.4 进程名显示

快照会尝试读取 exe 文件的 **FileDescription**（中文描述名），例如：

| exe 文件名 | 显示名称 |
|-----------|---------|
| explorer.exe | Windows 资源管理器 |
| Taskmgr.exe | 任务管理器 |
| rider64.exe | rider64.exe（无描述时显示原名） |

## 8. 参数配置

### 8.1 运行时可调参数

| 参数 | 位置 | 范围 | 默认值 | 说明 |
|------|------|------|--------|------|
| **采样间隔** | 底部面板输入框 | > 0 秒 | 1.0 秒 | 数据采集频率。增大间隔可进一步降低 CPU 开销 |
| **告警阈值** | 底部面板 SpinBox 或曲线图拖动 | 1–100% | 90% | CPU 超过此值时触发告警 |

修改参数后**立即生效**，无需重启监控。

### 8.2 代码级配置

编辑 `config.py` 中的 `AppConfig` 数据类：

```python
@dataclass
class AppConfig:
    interval: float = 1.0          # 采样间隔（秒）
    threshold: float = 90.0        # 告警阈值（%）
    max_data_points: int = 300     # 曲线最多保留数据点数
    log_cooldown: float = 30.0     # 告警最小间隔（秒）
    log_dir: str = "logs"          # 日志输出目录
```

## 10. 技术原理

### 10.1 CPU 数据源

本应用使用 **Windows PDH（Performance Data Helper）API** 采集 `% Processor Utility` 计数器，而非 psutil 默认的 `% Processor Time`。

| 指标 | % Processor Time (psutil) | % Processor Utility (PDH) |
|------|---------------------------|--------------------------|
| 计算方式 | 基于 CPU 时间片占用 | 基于 CPU 频率利用率 |
| 是否反映变频 | ❌ 不反映 | ✅ 反映 Turbo Boost |
| 与任务管理器一致 | ❌ 差距可达 3~20 倍 | ✅ 完全一致 |

### 10.2 进程 CPU 归一化

由于 PDH 不提供进程级数据，进程 CPU 仍使用 psutil 采集。为确保进程累加值等于总体 CPU，采用**按比例分配法**：

```
进程 CPU = (进程原始值 / 所有进程原始值之和) × PDH 总体值
```

这保证了：
- 各进程之间的相对占比正确（来自 psutil）
- 绝对值与任务管理器一致（来自 PDH）
- 进程累加 = 总体 CPU（数学保证，零误差）

### 10.3 分级采集策略

| 路径 | 频率 | 操作 | CPU 开销 |
|------|------|------|----------|
| **常规路径** | 每秒 | 仅调用 PDH 获取总体 CPU | < 0.05% |
| **告警路径** | 超阈值且冷却期后 | 独立 1s 窗口采集 Top 10 进程 | 仅触发时 |
| **快照路径** | 手动触发 | 独立线程采集 Top 20 进程 | 仅触发时 |

常规路径**不遍历任何进程**，这是自身 CPU 占用极低的关键。

## 11. 常见问题 FAQ

### Q: 启动后第一个数据点为什么偏低？

PDH 计数器需要两次 `PdhCollectQueryData` 之间的时间差来计算。首次调用是基线建立，第二次才有有效值。所以第一个数据点可能接近 0，这是正常现象。

### Q: 应用自身 CPU 占用是多少？

常规路径下约 **0.05%–0.15%**。即使触发告警（遍历进程列表），由于有 30 秒冷却期限制，整体开销也远低于 0.5%。

### Q: 进程 CPU 累加值为什么精确等于总体 CPU？

这是"按比例分配法"的数学保证。所有进程按原始值在总量中的占比分配 PDH 总体值，累加结果天然等于总体值。

### Q: 非 Windows 系统能用吗？

可以运行，但会自动降级到 psutil 数据源（`% Processor Time`），数据可能与系统监控工具不完全一致。快照功能的中文进程描述名也不可用。

### Q: 告警日志太多怎么办？

- 日志保存在 `logs/` 目录，可手动清理旧文件
- 界面告警列表最多显示最近 20 条
- 可通过调高告警阈值或增大 `log_cooldown` 减少告警频率

### Q: 如何调低告警阈值以便测试？

两种方式：
1. 底部面板的阈值 SpinBox 直接调整
2. 鼠标拖动曲线图上的红色虚线

修改即时生效，适合快速测试告警功能。

### Q: CPU 数值颜色含义？

| 颜色 | CPU 范围 | 含义 |
|------|---------|------|
| 🟢 绿色 | < 60% | 正常 |
| 🟠 橙色 | 60%–89% | 偏高 |
| 🔴 红色 | ≥ 90% | 告警 |

### Q: 关闭窗口时需要先停止监控吗？

不需要。直接关闭窗口即可，应用会自动断开信号、停止工作线程、释放 PDH 资源。
